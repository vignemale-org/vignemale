# Notes — Encore's Rust runtime (and implications for Vignemale)

Sources: https://encore.dev/blog/rust-runtime · benchmarks https://encore.dev/blog/ai-benchmark
(read on June 12, 2026)

## Why Rust (and not a sidecar)

- A Go runtime prototype as a **sidecar** (separate process, IPC) added **2-4 ms
  per request** in serialization + context switch alone, before any real work —
  abandoned. → the runtime must be **in-process** with the host language.
  Us: PyO3 in-process, same choice. ✅
- Rust = a reusable multi-language base (cf. prost, pydantic-core). A future
  Vignemale JS SDK would reuse the same core (our bet: an ORM driven by JSON
  descriptors, already language-agnostic).
- Single-threaded Node → everything non-business (HTTP cycle, DB pool, pubsub,
  trace) runs **multi-threaded on tokio**. Python's GIL plays the role of Node's
  single thread; we already release the GIL (`allow_threads`) on sqldb, and the
  handlers run in `spawn_blocking`.

## Architecture (what we already reproduce)

- **2 protos**: *App Metadata* (compile-time, generated by the parser) +
  *Runtime Config* (deploy-time, chooses the provider). We have exactly that:
  `collect` → meta.proto; `VIGNEMALE_*` env vars set at deploy. ✅
- The runtime **never parses** the code — it receives a structured description
  (collect lives in the CLI, not the runtime). ✅
- `CancellationGuard`: client disconnected → the handler finishes in a background
  task to emit the end of the trace. Equivalent on our side (timeout → the
  handler continues in the background, logs preserved).
- 67,077 lines of Rust on Encore's side (core + JS bindings + tsparser +
  supervisor).

## Pingora gateway — THE next building block

- Encore **embeds Pingora in the runtime process** (Cloudflare's HTTP proxy),
  not a separate proxy → zero serialization at the boundary.
- Provides: path routing to services, **CORS**, **HTTP/2**, connection pooling,
  **graceful draining** — natively.
- **The auth handler runs INSIDE the gateway**: Pingora calls the handler,
  retrieves `{user_id, auth_data}`, then forwards to the backend service with the
  propagated identity. This is our exact model (auth in the core + signed svcauth)
  — we're just missing the *gateway facade* that routes to N services.
- Encore had to add Windows support to Pingora (contributed upstream); us:
  Linux/macOS (containers) are enough.

Implication: today our multi-service setup runs in 1 process (calls =
functions) or in signed service-to-service HTTP. The gateway is the piece that, in
prod, **receives public traffic, authenticates at the edge, routes/forwards to
the right service** — the single entry point of a deployed app.

⚠️ Pingora is heavy (depends on boringssl/pingora-core, a large tree). Pragmatic
starting option: a gateway in **axum** (reuses all our HTTP stack, CORS, auth,
svcauth, trace already written); switch to Pingora if a real need justifies it
(upstream HTTP/2, fine-grained pooling). What matters is the gateway *function*,
not the library — the interface stays the same.

## Benchmarks (methodology to copy honestly)

- Tool: **oha**, 150 concurrent workers, 10 s, best-of-5.
- Two scenarios: simple JSON **and** with validation (the gap explodes there,
  because Encore validates at the Rust level via the parser's types).
- Encore.ts: 121k req/s simple, 107k validation, P99 2.3/3.6 ms.
  Express+Zod: 15.7k / 11.9k, P99 11.9/18.2 ms → "9× Express, -80% latency".
- **Our target = FastAPI/uvicorn** (direct Python competitor). Honesty: the
  handler stays Python (GIL), we won't beat Encore.ts; but the HTTP cycle, the
  routing, the query/header validation and the streaming run in Rust while the
  handler releases the GIL. Angle: show where Vignemale wins (I/O concurrency,
  streaming, trivial requests) and stay transparent where it's at parity (pure
  CPU in the handler).

## Ideas noted for later

- Custom varint trace (EventBuffer), more compact than protobuf; but OTel is
  requested by customers from the start → we aim for OTel.
- Sampling decided **at the start of the request**, propagated to all child spans
  (otherwise invalid partial traces) — to remember for our W3C trace.
- Structured errors (topic name, msg size, provider) > generic `anyhow::Context`
  — debt we can avoid by typing our sqldb/api errors.
